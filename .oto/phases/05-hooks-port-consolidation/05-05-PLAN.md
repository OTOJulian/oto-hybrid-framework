    // Phase 5 — implemented in Wave 4 (plan 05-05)
    // Covers: HK-01 snapshot lock per D-09 / ADR-04 Consequences.

    const assert = require('node:assert/strict');

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `oto/hooks/oto-session-start` stdout → fixture file | One-time capture into a static file; the file becomes the source of truth for regression testing. The boundary is gated by the hand-eyeball step in Task 1 (D-09 explicit). |
| Fixture file → test runtime | Read by `node:test`; comparison is structural (`deepEqual`) and substring (`indexOf`); no eval, no filesystem mutation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05-01 | Spoofing | A future change to `oto-session-start` reintroduces upstream identity strings; the regenerated fixture would carry the leak | mitigate | The test does TWO orthogonal checks: (1) deep-equal against the locked fixture, AND (2) substring scan inside the captured stdout for 6 banned strings. Even if a future hand-edit regenerates both the fixture AND the hook in lockstep, the substring scan catches `superpowers`/`gsd-`/etc. before the commit lands. ASVS L1 V14.3.4: defense in depth on identity-content invariants. |
| T-05-05-02 | Tampering | The fixture file lives in `oto/hooks/__fixtures__/`, which is INSIDE the package distribution (`oto/` is in package.json `files`). End users receive the fixture too. | accept | Fixture is small (~1KB) and inert. Personal-use cost ceiling: tiny per-install bytes are fine; Phase 10 may revisit moving fixtures out of the install payload. |
| T-05-05-03 | Tampering | tests/05-session-start-fixture.test.cjs spawns bash on a path under REPO_ROOT | mitigate | The hook path is constructed via `path.join(REPO_ROOT, 'oto', 'hooks', 'oto-session-start')` — fixed string, no user input. Spawned via `spawnSync('bash', [HOOK])` with `shell: false` (default), so even a malicious file content can't shell-out without a deliberate `system()` call inside the bash itself (which is part of the threat model from plan 05-03, already mitigated there). |
| T-05-05-04 | Information Disclosure | Test stdout/stderr could leak PATH or HOME if spawnSync error path triggers | accept | `spawnSync` env explicitly passes `PATH` and `HOME` (needed by bash). Failure mode shows error message including these env vars in test output. Personal-use scope; no CI exposure (Phase 10 will set explicit env in CI). |
| T-05-05-05 | Repudiation | Fixture drift could be silently committed if the tester runs `git add oto/hooks/__fixtures__/session-start-claude.json` blindly | mitigate | The test runs on every `npm test`; it fails LOUDLY when the hook output drifts. The hand-eyeball gate in Task 1 (D-09) is the explicit "review before commit" step; failing the test means re-running Task 1's capture + hand-eyeball cycle. |
</threat_model>
