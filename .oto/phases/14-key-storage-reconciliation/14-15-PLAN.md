Close Gap 1 from 14-VERIFICATION.md (2026-07-12): both `cmdConfigNewProject` (oto/bin/lib/config.cjs:186) and `configNewProject` (sdk/src/query/config-mutation.ts:355) accept ANY JSON root as choices. `JSON.parse('[{"exa_search":"<key>"}]')` yields an array; object spread (`{...choices}`) copies its numeric index property into the merged config, so the plaintext key persists at `0.exa_search` in git-tracked config. `reconcileNewProjectIntegrations` (both layers) inspects only the three TOP-LEVEL integration fields, so the nested plaintext sails through and both commands exit 0. The same hole admits nested plaintext through legitimate objects (e.g. `{"git":{"exa_search":"<key>"}}` survives the git sub-merge).

Two-layer fix, mirrored CJS/SDK:
1. **Shape guard:** parsed choices must be a non-null plain object (never array/primitive/null) — rejected immediately after JSON.parse, BEFORE the `.oto`/`.planning` mkdir side effect.
2. **Deep scan:** `findNestedIntegrationString` walks the fully merged config below the top level and rejects any `exa_search`/`brave_search`/`firecrawl` key holding a string, naming the dot-path but never the value. Runs FIRST inside `reconcileNewProjectIntegrations`, before its keyfile-write/defaults-heal side effects. (Top level stays with the existing loop: caller-owned strings reject per D-05, trusted global-default strings migrate per D-02.)
