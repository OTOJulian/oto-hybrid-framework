CR-02: both `cmdConfigNewProject` (oto/bin/lib/config.cjs:186-223) and `configNewProject` (sdk/src/query/config-mutation.ts:355-484) accept ANY JSON root as choices. `JSON.parse('[{"exa_search":"<key>"}]')` yields an array; object spread copies its numeric index property into the merged config, so the plaintext key persists at `0.exa_search` in git-tracked config. Reconciliation inspects only the three TOP-LEVEL integration fields, so nested plaintext sails through and both commands exit 0. The same hole admits nested plaintext through legitimate objects (`{"git":{"exa_search":"<key>"}}` survives the git sub-merge), through unknown top-level keys (`{"exa_searchh":"<key>"}`), and ‚Äî per the review ‚Äî the empty string `""` must also reject (a nested `""` on an integration key is never legitimate config).

CR-02 has a fourth face, the PARSE-ECHO hole (verified against live source): both parse catches interpolate the parse error ‚Äî CJS `error('Invalid JSON for config-new-project: ' + err.message)` at config.cjs:202, SDK `const msg = err instanceof Error ? err.message : String(err); throw new GSDError(\`Invalid JSON for config-new-project: ${msg}\`, ...)` at config-mutation.ts:369-370. Modern V8 JSON.parse error messages quote input snippets (e.g. `Unexpected token 'b', "{bad "exa_"... is not valid JSON`), so malformed choices JSON that contains a key echoes key bytes back to stdout/stderr ‚Äî contradicting this plan's "value never echoed" truth. Both catches must use a FIXED message.

WR-06: defaults and caller choices are merged BEFORE reconciliation, and the one-pass loop writes keyfiles as it encounters values. Two concrete failures: (a) an explicit caller boolean hides a legacy default string so it is never migrated (plaintext stays in ~/.oto/defaults.json unhealed and provenance is lost); (b) a valid Exa default is written to a keyfile before a later invalid Brave caller value rejects the command ‚Äî side effects from a failed operation.

Side-effect ordering contradiction (required amendment 2): verified against live source ‚Äî CJS mkdirs `.oto` at config.cjs:206-213 BEFORE `buildNewProjectConfig` (which validates) at :215; SDK mkdirs at config-mutation.ts:374-378 BEFORE reconcile at :471. **Resolution chosen: the PREFERRED FIX ‚Äî move directory creation until after complete validation.** This is feasible in both layers: `buildNewProjectConfig` does not require the project directory to exist (it only reads env/home paths), and the SDK's defaults read + merge + reconcile have no dependency on `paths.planning` existing.
