<objective>
Close Gap 1 from 14-VERIFICATION.md (CR-02) and WR-06 from 14-REVIEW.md, fully.

CR-02: both `cmdConfigNewProject` (oto/bin/lib/config.cjs:186-223) and `configNewProject` (sdk/src/query/config-mutation.ts:355-484) accept ANY JSON root as choices. `JSON.parse('[{"exa_search":"<key>"}]')` yields an array; object spread copies its numeric index property into the merged config, so the plaintext key persists at `0.exa_search` in git-tracked config. Reconciliation inspects only the three TOP-LEVEL integration fields, so nested plaintext sails through and both commands exit 0. The same hole admits nested plaintext through legitimate objects (`{"git":{"exa_search":"<key>"}}` survives the git sub-merge), through unknown top-level keys (`{"exa_searchh":"<key>"}`), and — per the review — the empty string `""` must also reject (a nested `""` on an integration key is never legitimate config).

WR-06: defaults and caller choices are merged BEFORE reconciliation, and the one-pass loop writes keyfiles as it encounters values. Two concrete failures: (a) an explicit caller boolean hides a legacy default string so it is never migrated (plaintext stays in ~/.oto/defaults.json unhealed and provenance is lost); (b) a valid Exa default is written to a keyfile before a later invalid Brave caller value rejects the command — side effects from a failed operation.

Side-effect ordering contradiction (required amendment 2): verified against live source — CJS mkdirs `.oto` at config.cjs:206-213 BEFORE `buildNewProjectConfig` (which validates) at :215; SDK mkdirs at config-mutation.ts:374-378 BEFORE reconcile at :471. **Resolution chosen: the PREFERRED FIX — move directory creation until after complete validation.** This is feasible in both layers: `buildNewProjectConfig` does not require the project directory to exist (it only reads env/home paths), and the SDK's defaults read + merge + reconcile have no dependency on `paths.planning` existing. After this plan, "invalid input leaves `.oto/` absent" is a satisfiable acceptance test in both layers.

The fix, mirrored CJS/SDK:
1. **Shape guard:** parsed choices must be a non-null plain object — rejected immediately after JSON.parse.
2. **Key allowlist:** every top-level choice key must be a known new-project key (derived from the hardcoded defaults object plus the documented new-project-only keys `mode`, `granularity`, `depth`); unknown keys reject naming the KEY only.
3. **Two-phase reconcile with provenance:** Phase A validates the deep-scanned merged config, caller-owned integration values, and RAW legacy defaults with zero side effects. Phase B migrates default strings to 0600 keyfiles (keyfile-wins on conflict, D-02) with rollback compensation, then finalizes config booleans — caller booleans always win in config, default strings always migrate.
4. **mkdir after validation** in cmdConfigNewProject, ensureConfigFile, and configNewProject.

Scope note: NO sdk/dist rebuild here — plan 14-19 owns the phase's single rebuild and re-runs the verifier's reproductions against dist.

Purpose: SECR-01/SECR-02 — no integration API key can be persisted in committed config through either write path, at any nesting depth, under any key spelling, and no failed new-project run leaves partial side effects.
Output: shape guards + key allowlists + two-phase reconcilers in both layers, real-process regression suites.
