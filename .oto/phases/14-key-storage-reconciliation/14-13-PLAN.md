<objective>
Close Gap 2 from 14-VERIFICATION.md (2026-07-12): `readKeyfile` in both layers returns trimmed-empty content as a present keyfile (`{ value: '', healed }`). With a valid legacy string in config and a zero-byte/whitespace keyfile on disk, `migrateLegacyIntegrationKeys` evaluates `existing.value !== current` → true, takes the keyfile-wins conflict branch (D-02), drops the only valid credential, writes `true` to config, and leaves the keyfile empty. `secretStatus` then reports `Exa: enabled — key from ~/.oto/exa_api_key ((unset))`.

Fix: a trimmed-empty keyfile is not a credential — treat it as absent in `readKeyfile` (the single choke point both layers' migration, detection, warning, and status paths flow through). Migration then overwrites the empty file with the legacy value via `writeKeyfile`, which already enforces mode 0600.

Purpose: SECR-03's self-heal must never destroy the only stored credential; SECR-04's status must never report an unset key as usable.
Output: two-line guarded fix in each layer's `readKeyfile` + CJS and SDK regression suites for zero-byte and whitespace-only files.
</objective>

      if (!fs.existsSync(target)) return null;

      const value = fs.readFileSync(target, 'utf8').trim();
