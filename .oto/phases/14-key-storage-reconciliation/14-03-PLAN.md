<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| terminal/stdin → secret-set handler | the sanctioned key-entry channel |
| shell argv → oto-sdk query | hostile-by-default: argv leaks to shell history, `ps`, and agent transcripts |
| handler output → stdout/JSON consumers | anything returned may land in logs or Claude's context |
| process → ~/.oto keyfiles | secret material at rest |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-16 | I | key passed as argv to secret-set (history/ps/transcript leak) | mitigate | hard reject when args[1] present; stdin/TTY are the only input channels (D-09) |
| T-14-17 | I | key echoed while typing at the TTY prompt | mitigate | readline with muted `_writeToOutput` — no echo; prompt on stderr |
| T-14-18 | I | raw key in handler return data / secret-status output | mitigate | results carry maskSecret values only; test asserts serialized results lack the raw key |
| T-14-19 | I | keyfile written with default umask permissions | mitigate | writeKeyfile 0600 + chmod (Plan 02 helper); secret-status heals drift (D-12) |
| T-14-20 | T | slug-driven path traversal (`secret-set ../../x`) | mitigate | resolveSlug allowlist; keyfilePath throws on unknown slug |
| T-14-21 | R | user believes a key is cleared while EXA_API_KEY still supplies it | mitigate | secret-clear reports `env still set — integration remains available` (D-11) |
| T-14-22 | E | smoke test leaving a real-looking key + true flag committed in repo config | mitigate | smoke uses a labeled dummy value, ends with secret-clear + `git diff --exit-code .oto/config.json` gate |
</threat_model>
