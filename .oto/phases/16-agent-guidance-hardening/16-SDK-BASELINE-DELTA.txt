At `ce91d4d^` the old assertions pass while the new Phase 16 keyfile tests fail; at `ce91d4d` all five credential-resolution tests pass while these two old expectations fail. These are stale pre-Phase-16 expectations, not production-code regressions. They remain unedited pending developer approval.

The SDK correctly reports one pending todo; legacy CJS still scans `.planning` and reports zero. Temporarily removing only that todo makes both assertions pass. This is the tracked WR-02 planning-root debt family, not a Phase 16 production-code regression. Fixing the parity in production would begin the separately scoped planning-root migration; hiding it in the golden test would weaken a real parity check. Neither action was taken.
