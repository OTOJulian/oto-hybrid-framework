## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| installer -> user filesystem (~/.local/bin) | trySelfLinkOtoSdk writes a link/wrapper into a user-owned PATH dir |
| prior install -> current install | a stale oto-sdk target may already exist from an earlier version |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-07 | Elevation of Privilege | trySelfLinkOtoSdk write target | mitigate | Restrict targets to `~/.local/bin` and on-PATH dirs resolving UNDER `os.homedir()` only; skip win32; non-fatal on EROFS/EACCES (try next / return null). Faithful port preserves this bounding. Unit test asserts the target lands under the stubbed HOME. |
| T-11-08 | Tampering | stale / poisoned prior `oto-sdk` link | mitigate | UNLINK any existing target before creating the new symlink/wrapper (preserve the dead source's unlink-then-create). Unit test asserts a pre-existing stale target is replaced, not appended-to or errored on. |
| T-11-09 | Spoofing | false "OTO SDK ready" when bin not callable | mitigate | Gate the ready message on isOtoSdkOnPath() AFTER attempting the self-link (Pitfall 3) — never on dist-file presence alone. Acceptance asserts the string lives only in the onPath-true branch. |
</threat_model>
