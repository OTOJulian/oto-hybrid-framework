**## Context** — Phase 15 registers the Exa MCP server in three runtimes; the key lives only in `~/.oto/exa_api_key` (0600) or `EXA_API_KEY` (Phase 14). The transport determines whether key material must appear in runtime config files.

**## Decision** — state as fact:
1. Transport is **launcher-stdio**: a shipped launcher `oto/hooks/oto-exa-mcp.js` (distributed via the existing `scripts/build-hooks.js` channel) resolves the key from env/keyfile at spawn time and spawns the server. No key material in any runtime config file (D-01).
2. Exact pin: `npx -y exa-mcp-server@3.2.1` (D-02). Version bumps are one-line reviewed diffs.
3. Tool surface pinned via the Smithery positional arg `tools=web_search_exa,web_fetch_exa,web_search_advanced_exa` — NOT `--tools` flags (the 3.2.1 stdio bundle parses only positional `key=value` args; `web_search_advanced_exa` is off by default, so the arg is REQUIRED for MCP-06's 3-tool surface).
4. Pre-warm mechanism (D-03, corrected from the illustrative `--version`): the 3.2.1 binary has no `--version` handling and would hang on stdio. Pre-warm = empty-stdin spawn with timeout: `spawnSync('npx', ['-y','exa-mcp-server@3.2.1','tools=…'], { input: '', timeout: 120000 })` — server exits 0 on stdin EOF (verified against the real bundle). Pre-warm failure warns at install time but does NOT block registration (consent was given; the launcher surfaces the same failure per-session and npx succeeds once online).
5. D-15 key-usability rule, as a table applied identically in all three implementation sites (CJS `oto/bin/lib/secrets.cjs`, SDK `sdk/src/query/secrets.ts`, launcher inline copy):

| Step | Rule |
|------|------|
| 1 | `EXA_API_KEY` env var, trimmed — non-empty wins |
| 2 | Keyfile `~/.oto/exa_api_key`: `stat` (symlinks FOLLOWED) must report a regular file |
| 3 | Read UTF-8, trim — non-empty is usable; empty/whitespace = "no key detected" |
| 4 | Non-regular targets (dir, FIFO, dangling symlink) = "no key detected" |

Record the reconciliation with Phase 14 WR-07 explicitly: the READ/DETECT path follows symlinks to regular files (legit use case: symlink into a password-manager-managed file; read-follow is disclosure-neutral for the same user); the WRITE path keeps lstat + O_NOFOLLOW refusal (the WR-07 threat was write-through-symlink).

**## Rationale** — uniform secret indirection across all three runtimes; single rotation point in the Phase 14 keyfile; one tool-pinning mechanism instead of three per-runtime filters; deterministic tool surface under an exact pin (upstream already deprecated 5+ tools).

**## Consequences** — includes: a Node child-process hop per session; version bumps require a release; the **evaluated alternative — remote HTTP** (`https://mcp.exa.ai/mcp` + `x-api-key` header) is documented here with trade-offs (D-04): zero child process, Exa-maintained, offers a keyless free tier (~150 calls/day — this anchors the EXA-F-01 v0.5.x+ deferral) — rejected because it requires a literal key in 2-3 user-private config files (rotation pain), Gemini headers cannot expand env vars (gemini-cli#5282), and Codex `bearer_token_env_var` sends the wrong header scheme.

Do NOT use the words "v1", "simplified", or "for now" anywhere.
