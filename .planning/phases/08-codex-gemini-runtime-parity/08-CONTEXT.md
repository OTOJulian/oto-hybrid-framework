# Phase 8: Codex & Gemini Runtime Parity - Context

**Gathered:** 2026-05-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Bring Codex and Gemini CLI runtimes up to functional parity with Claude. Three deliverables, all bounded by ROADMAP Phase 8 and REQUIREMENTS MR-02..04:

1. **Single source-of-truth (SoT) template** for the project-root instruction files (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`) plus per-runtime transformations ‚Äî drift-impossible by construction (MR-02 / SC#1).
2. **`decisions/runtime-tool-matrix.md`** ‚Äî auto-generated divergence document (tool name maps, frontmatter dialects, capability gaps, hook surfaces, sandbox modes) plus a per-command runtime-support table ‚Äî verified by per-runtime golden-file fixture tests (MR-03 / SC#2).
3. **Per-runtime smoke tests** ‚Äî install ‚Üí run a representative spine of `/oto-*` commands ‚Üí state file written ‚Üí no "tool not found" errors (MR-04 / SC#3).

**User-locked bar (overrides "best-effort" framing in roadmap):** No commands that don't work on Codex; no commands that don't work on Gemini. Both runtimes are daily-peer to Claude. The "best-effort given subagent-support gaps" language in ROADMAP Phase 8 is **superseded by user direction in this phase** ‚Äî gaps must be closed by transforms (e.g. Task() ‚Üí Gemini agent-as-tool rewrite at install time), not documented as accepted limitations.

What Phase 8 does **not** ship (out of scope, deferred):
- Upstream-sync handling for new Codex/Gemini transforms appearing in upstream ‚Äî Phase 9 owns sync mechanics.
- Full CI promotion of the per-runtime smoke tests ‚Äî Phase 10 owns CI.
- Three-way merge UX or sync v2 ‚Äî Phase 9/v2.
- A separate per-runtime stability-gate equivalent of MR-01 for Codex/Gemini ‚Äî recorded as deferred (no requirement currently mandates it; Phase 8 closeout demonstrates daily-peer functionality through the spine smoke + fixture tests).

## Implementation Decisions

### SoT Template for Instruction Files (MR-02 / SC#1)

- **D-01:** SoT governs the **three project-root instruction files** (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`) that each runtime reads when operating inside this repo. The install-time marker block (the ~4-line oto bootstrap injected by `renderInstructionBlock(ctx)` into the user's `~/.<runtime>/<FILE>`) stays as small per-adapter strings ‚Äî it's tiny, user-owned, and not part of the MR-02 surface. SC#1's "no hand-edits to the runtime files" applies to the project-root files.

- **D-02:** SoT artifact = **single markdown template with embedded runtime-tag fences**. File: `oto/templates/instruction-file.md`. Per-runtime divergences expressed as `<!-- runtime:codex -->...<!-- /runtime:codex -->`, `<!-- runtime:gemini -->...<!-- /runtime:gemini -->` (and optionally `<!-- runtime:claude -->...<!-- /runtime:claude -->` for Claude-only sections). Tokens of the form `{{token}}` for any computed values (e.g. `{{otoVersion}}`, `{{repoUrl}}`). A small `bin/lib/instruction-file.cjs` exports `render(runtimeName, ctx)` that strips the wrong-runtime fences and substitutes tokens. Diff-friendly, grep-friendly, single file to edit.

- **D-03:** **"Drift impossible by construction" enforcement = generated files committed + node:test regen-diff check.** The three project-root files (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`) are generated by `scripts/render-instruction-files.cjs` and **committed to the repo** (so contributors and runtimes can read them statically from a fresh clone, no build step needed). A `tests/phase-08-instruction-file-render.test.cjs` runs the renderer in-memory, diffs the output against the committed files, and fails if they differ. Any hand-edit to the generated files fails the test on next run; the only way to update them is to edit the template + run the regen script.

- **D-04:** **Seed content for the SoT template = existing project-root `CLAUDE.md`** (`/Users/Julian/Desktop/oto-hybrid-framework/CLAUDE.md`, ~100+ lines: project description, tech stack TL;DR, GSD workflow enforcement, etc.). The template body is the current CLAUDE.md verbatim. Runtime-tag fences are added only where Codex and Gemini diverge:
  - Codex section: sandbox-mode notes, AGENTS.md naming explanation, codex-config.toml hook block reference.
  - Gemini section: subagent-fallback note (Task() rewrites at install time), `${VAR}`‚Üí`$VAR` body-escape note, hook surface (or gap) per researcher findings.
  - Shared section (no runtime tag): everything else ‚Äî project description, tech stack, conventions, GSD workflow enforcement, project skills, developer profile.

  After D-04 lands, project-root `CLAUDE.md` is the generated output of itself (the template), `AGENTS.md` and `GEMINI.md` are newly generated.

### `runtime-tool-matrix.md` Source and Scope (MR-03 / SC#2)

- **D-05:** **Auto-generated** from a small JS source-of-truth ‚Äî `bin/lib/runtime-matrix.cjs` (or merged into `bin/lib/instruction-file.cjs` if convenient) reads:
  - The three `bin/lib/runtime-*.cjs` adapter descriptors (configDirEnvVar, instructionFilename, settingsFilename, settingsFormat, agentSandboxes for Codex, sourceDirs/targetSubdirs).
  - The Claude‚ÜíGemini tool map (lifted from upstream `bin/install.js` line 955 area, now living in oto's `bin/lib/gemini-transform.cjs` per D-13).
  - The capability-gap registry (a small inline JS table: e.g. `{ gemini: { subagentDispatch: 'inline-rewrite via agent-as-tool', hooks: '...' }, codex: { sandboxEnforcement: 'per-agent .toml required', ... } }`).

  A `scripts/render-runtime-matrix.cjs` emits `decisions/runtime-tool-matrix.md`. Same regen-diff test pattern as D-03 (test re-runs, diffs against committed file, fails on drift).

- **D-06:** **Matrix dimensions covered (all four):**
  1. **Tool name map** (Claude ‚Üí Codex, Claude ‚Üí Gemini) including the explicit map (Read‚Üíread_file, Bash‚Üírun_shell_command, ‚Ä¶) and filtered tools (`mcp__*` excluded on Gemini, `Task` excluded on Gemini because agents auto-register as callable tools).
  2. **Frontmatter dialect per runtime** ‚Äî Claude (name/description/tools/color), Codex (no tools/color in frontmatter, `<codex_agent_role>` header injected, sandbox_mode in companion `.toml`), Gemini (tools as YAML array, no color, no skills, `${VAR}`‚Üí`$VAR` in body).
  3. **Capability gaps + 'best-effort' boundary** ‚Äî explicit list of what each runtime cannot do natively and what oto does to close the gap. With the user's daily-peer bar, this section primarily documents how each gap is closed (Task() inline-rewrite for Gemini, per-agent `.toml` for Codex, etc.) rather than what's left unsupported.
  4. **Hook event names + sandbox modes per runtime** ‚Äî Claude (settings.json events: SessionStart, PreToolUse, PostToolUse, Statusline, Stop), Codex (`[hooks]` TOML + per-agent `sandbox_mode` in `<configDir>/agents/<agent>.toml`), Gemini (per researcher findings; mirror Claude's settings.json shape if Gemini supports it).

- **D-07:** **`runtime-tool-matrix.md` lives in `decisions/`** alongside ADRs and `agent-audit.md` / `skill-vs-command.md`. Treats it as a normative reference (decision-level event to change). Mirrors precedent set by other decision-grade docs.

- **D-08:** **Per-runtime fixture verification** = **golden-file fixtures + diff tests**. For each runtime:
  - Pick a representative agent (`oto-executor`), a representative command (`/oto-progress`), and a representative skill (`oto:test-driven-development`) as the canonical fixture trio.
  - For Codex: pass each through `runtime-codex.cjs::transformAgent/transformCommand/transformSkill` and diff against `tests/fixtures/runtime-parity/codex/oto-executor.expected.md`, `oto-progress.expected.md`, `test-driven-development.expected.md`. Plus per-agent `.toml` fixture (`oto-executor.expected.toml`).
  - For Gemini: same trio, against `tests/fixtures/runtime-parity/gemini/`.
  - For Claude: identity transforms ‚Äî fixture diffs assert content unchanged (sanity baseline).
  - Plus a **matrix-vs-code consistency test**: assert that the rendered matrix's Claude‚ÜíGemini tool map matches the actual `convertGeminiToolName` table in `bin/lib/gemini-transform.cjs` byte-for-byte, and that the matrix's Codex sandbox-mode column matches `runtime-codex.cjs::agentSandboxes` byte-for-byte. Catches drift between matrix and code.

- **D-09:** **Per-command runtime support matrix** is generated as a section inside `decisions/runtime-tool-matrix.md`. A small scanner walks `oto/commands/**/*.md`, parses each command's referenced agents and tools, cross-checks against per-runtime capabilities, emits a table: `| /oto-* command | Claude | Codex | Gemini |`. **Codex column must be 100% green for v0.1.0 closeout.** Gemini column must be 100% green per user's daily-peer bar (D-12). Same regen-diff test enforces freshness.

### Codex Parity Depth (MR-02..04)

- **D-10:** **Full upstream parity port.** User intends to use Codex day-to-day. Bar: no commands that don't work on Codex. Phase 8 ports:
  - **`mergeSettings`** for `~/.codex/config.toml` ‚Äî registers oto's hooks under a marker block, mirroring the Claude pattern (Phase 5 D-12) but in TOML. Lands as `bin/lib/codex-toml.cjs::mergeHooksBlock(existingText, hookEntries) ‚Üí string`. Hand-rolled TOML (zero deps per CLAUDE.md / Pitfall 11). Reference: upstream `foundation-frameworks/get-shit-done-main/bin/install.js` lines 2168‚Äì2965.
  - **`generateCodexAgentToml`** ‚Äî emits one `<configDir>/agents/<agent>.toml` per retained agent, encoding `name`, `description`, `sandbox_mode` (from `runtime-codex.cjs::agentSandboxes`), `developer_instructions` (from agent body), and `model` / `model_reasoning_effort` (from profile resolution). Reference: upstream lines 1991‚Äì2059.
  - **`model_overrides`** ‚Äî per-agent model pins via `~/.oto/defaults.json` (oto's namespaced equivalent of upstream's `~/.gsd/defaults.json`; new env var space per ADR-02). Schema: `{ "model_overrides": { "oto-executor": "...", ... } }`. Read at install time, embedded into per-agent `.toml`.
  - **`RUNTIME_PROFILE_MAP` resolution** ‚Äî `/oto-set-profile` (already shipped Phase 4 WF-30) writes the active profile to `.oto/config.json`; Codex install reads it and resolves per-agent `(model, reasoning_effort)` from a profile-tier table.   - **Idempotent strip-and-rewrite** ‚Äî re-install detects oto's prior block via marker, strips it cleanly, rewrites with current state. Symmetric uninstall removes only oto-marked entries. Mirrors Phase 5 D-13/D-14 hook contract.

  Total estimated: ~600‚Äì800 LOC ported into `bin/lib/codex-toml.cjs` + `codex-profile.cjs` + transform updates in `runtime-codex.cjs`. Zero new dependencies.

- **D-11:** **Codex `transformAgent` ports `convertClaudeAgentToCodexAgent` verbatim** (upstream `bin/install.js` line 1991): preserves frontmatter `name` and `description`, drops `tools` and `color` from frontmatter (Codex-incompatible), injects `<codex_agent_role>` header with role/tools/purpose. The `transformCommand` and `transformSkill` for Codex are largely identity at this point (Codex reads markdown commands/skills natively); confirmed during planning research.

### Gemini Parity Depth (MR-02..04)

- **D-12:** **Daily-peer bar applies to Gemini too.** No commands that don't work on Gemini. The roadmap's "best-effort given subagent-support gaps in Gemini" framing is **superseded by user direction**. Gaps must be closed by transforms; documented gaps are not acceptable for v0.1.0 commands.

- **D-13:** **Gemini transforms port upstream `bin/install.js` patterns verbatim, into `bin/lib/gemini-transform.cjs`:**
  - **`convertGeminiToolName`** (line 955) ‚Äî exact map (Read‚Üíread_file, Bash‚Üírun_shell_command, etc.), `mcp__*` filtered (auto-discovered from mcpServers config at runtime), `Task` filtered (agents auto-registered as callable tools), default lowercase.
  - **`convertClaudeToGeminiAgent`** (line 3510) ‚Äî `tools` becomes YAML array (not comma-separated string), tool names converted via `convertGeminiToolName`, `color:` and `skills:` frontmatter fields stripped (Gemini validation errors), `${VAR}`‚Üí`$VAR` body escape (Gemini's `templateString()` chokes on `${word}` patterns).
  - **`convertClaudeToGeminiToml`** (line 3942) ‚Äî markdown command ‚Üí TOML (`description = "..."`, `prompt = "..."`). For Gemini, commands are TOML files in `<configDir>/commands/`.

- **D-14:** **Gemini Task() rewrite = inline at install/transform time.** `runtime-gemini.cjs::transformCommand` and `transformAgent` scan markdown bodies for `Task(subagent_type='oto-X', ...)` patterns and rewrite them into Gemini's agent-as-tool invocation syntax. **The exact Gemini invocation syntax is the highest-priority research input for the Phase 8 researcher** ‚Äî the planner cannot lock this until research confirms it. Single-source workflow markdown stays as `Task(...)` in the repo; Gemini sees its native form at install time. Mirrors how `convertClaudeToGeminiAgent` already escapes `${VAR}`.

- **D-15:** **Parallel Task() dispatch on Gemini** ‚Äî Phase 8 researcher must verify whether Gemini's agent-as-tool invocation supports parallel calls in one tool-use block. **If yes:** parallel `Task()` rewrites to parallel Gemini invocations; behavior matches Claude. **If no:** transform sequentializes parallel `Task()` blocks into one-invocation-per-turn on Gemini, and the matrix documents "parallel agents on Gemini run sequentially (slower than Claude/Codex)." Either way, the command works ‚Äî daily-peer bar honored.

- **D-16:** **Gemini hooks** ‚Äî Phase 8 researcher confirms whether Gemini's `settings.json` supports hook events analogous to Claude's. **If yes:** port Claude's `mergeSettings` shape to Gemini (probably near-identity since both use settings.json). **If no or partial:** stub the missing events explicitly and document the gap in the matrix; statusline / validate-commit / prompt-guard may be silent on Gemini if Gemini doesn't fire those events. This is the one place D-12's "no broken commands" bar bends ‚Äî hooks are not commands; their absence degrades observability/safety on Gemini, doesn't break command execution.
