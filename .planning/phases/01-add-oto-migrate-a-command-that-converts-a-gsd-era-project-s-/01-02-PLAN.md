<objective>
Implement the migration engine module `oto/bin/lib/migrate.cjs` that converts a GSD-era user project to oto's planning surface. Reuses the existing `scripts/rebrand/lib/engine.cjs` for in-file string rewrites — with two small, additive options (`skipReports`, `reportsDir`) so the engine does not write to a non-writable install dir when invoked from a globally-installed binary. Adds migrate-specific logic for detection, marker-block rewrite, frontmatter-key rewrite, conflict detection, backup, idempotency, and CLI dispatch. Builds a runtime-derived rename map that excludes the canonical `.planning → .oto` path rule, making the directory rename explicitly opt-in (`--rename-state-dir`) instead of structurally unavoidable.

Purpose: This is the core workhorse. It delivers REQ-MIG-02 through REQ-MIG-08 — every behavioral requirement except the surface (CLI dispatch case + Claude command markdown) which Plan 03 handles.

Output:
- A small, additive change to `scripts/rebrand/lib/engine.cjs` (skipReports/reportsDir options) plus its unit-test coverage in `tests/rebrand-engine.test.cjs`.
- A new file `oto/bin/lib/migrate.cjs` (~300-400 LOC) with full test coverage (Tests 2-8 from Plan 01 turn green).

CRITICAL engine constraints (verified at engine.cjs:296-335 and engine.cjs:80-99):
1. `applyTree` requires `target !== out` (engine wipes `out` if non-empty and force:true).
2. `resolveOut` only allows out paths under repoRoot OR os.tmpdir(); migrate must therefore use a tmpdir staging path, then atomic-rename/copy back to projectDir.
3. `loadAndValidate` reads `repoRoot()/schema/rename-map.json` for validation; this is auto-resolved relative to engine's __dirname, NOT the user's project. The validator runs against whichever mapPath the caller provides — so a derived migrate-scoped map written to a temp file will validate against the same schema.
4. **B-04:** `applyTree` (engine.cjs:320) unconditionally calls `ensureReportsDir()` which writes to `repoRoot()/reports`. After global install, repoRoot() resolves to `~/.npm/lib/node_modules/oto/...` which is typically not user-writable → mid-apply EACCES. Task 0 fixes this with `skipReports` / `reportsDir`.

The rebrand engine handles these on .planning/**/*.md content via standard rule application:
- identifier: `gsd` → `oto`, `Get Shit Done` → `oto`, `gsd_state_version` → `oto_state_version`, `gsd-hook-version` → `oto-hook-version`, `GSD_VERSION` → `OTO_VERSION`, `superpowers` → `oto`
- path: `.planning` → `.oto` (segment) — **B-01: this rule MUST be filtered out of the migrate-derived map; the directory rename is opt-in via `--rename-state-dir`, applied by migrate AFTER engine string rules**
- path: `commands/gsd/` → `commands/oto/` (prefix), `agents/gsd-` → `agents/oto-` (prefix) — these are kept; user projects do not typically contain these path prefixes, but if they do they should rename
- command: `/gsd-` → `/oto-`
- env_var: `GSD_*` → `OTO_*`

What migrate adds on top of the engine:
- `<!-- GSD:<section>-start [source:<file>] -->` → `<!-- OTO:<section>-start [source:<file>] -->` marker rewrite. The engine's identifier rule `gsd → oto` would match the `GSD` inside the comment but it would also match `GSD` in narrative prose. Safer: explicit marker-block rewrite first, then run engine.
- Frontmatter-key rewrite for STATE.md is covered by the identifier rule (`gsd_state_version` → `oto_state_version`); migrate adds an idempotency-safe pass that targets only the YAML frontmatter slice as a defense-in-depth measure.

      const entries = fs.readdirSync(src, { recursive: true, withFileTypes: true });
      for (const entry of entries) {
        if (!entry.isFile()) continue;
