<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local working tree → tag | Pushing a tag against a not-yet-pushed commit creates an orphaned tag pointing at a SHA only present locally; remote release.yml will fail to fetch (T-REL-01) |
| Tag → release.yml validate job | Mismatched package.json.version vs tag breaks release.yml validate job; if the validate job is skipped (e.g. permissions misconfig), bad release ships (T-REL-01) |
| GitHub Release artifact → end-user `npm install` | If the tagged commit's source tarball is missing files from `package.json` `files` allowlist, end users get a broken install (T-INST-01) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-REL-01 | Tampering | git tag v0.1.0 vs origin/main | mitigate | Task 1 step 6 pushes the version-bump commit BEFORE Task 2 creates the tag. Task 2 step 1 explicitly verifies `git rev-parse @{upstream} == git rev-parse HEAD` before tagging. release.yml validate job double-checks `tag === package.json.version` server-side. Per W-3, on structural failure the tag is preserved (NOT deleted) — recovery is always-forward via v0.1.1, which prevents tag-rewrite races. |
| T-INST-01 | Tampering / DoS | tarball contents at v0.1.0 SHA | mitigate | install-smoke.yml runs as a workflow_call gate inside release.yml — if the tarball is missing files, the smoke job fails and `gh release create` never runs. Per W-5, local install-smoke uses exponential backoff (5s/15s/30s) and CI install-smoke.yml is the authoritative gate. Task 3 manual UAT installs from the actual release archive on a clean tmpdir as final acceptance. |
